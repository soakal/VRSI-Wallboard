## HIGH

### H1 — File-upload import can wipe the board with no guard
`server/src/routes/board.ts:217-230` · `server/src/services/boardService.ts:354-368` · `server/src/storage/localProvider.ts:222`

The JSON import path rejects an empty result (`no_valid_jobs`, board.ts:240), but the **file-upload path calls `applyBoardImport` unconditionally** — even when `parseXlsm` returns zero jobs (wrong file, or a schedule whose header row changed so the job-number column isn't detected). Zero jobs → `saveJobsFile([])` runs `DELETE FROM jobs` and inserts nothing → `pruneOrphanedBoardState(new Set())` deletes every board-state row that has no notes and isn't blocked. One wrong upload silently erases all manual statuses, binder checkmarks, and ship-date overrides on note-less jobs; recoverable only by restore-merge from a backup.
**Fix:** mirror the JSON guard — reject the file path with 400 when `jobs.length === 0`; optionally refuse an import that would drop more than N% of existing jobs without an explicit `force` flag.

### H2 — Writable scripts + elevated backup task = local privilege escalation
`scripts/windows/Install-WallBoard.ps1:231` · `scripts/windows/Register-BackupTask.ps1:28`

The installer grants the kiosk user `Modify` recursively over the **entire install tree** (`icacls "$RepoRoot" /grant "<user>:(OI)(CI)M" /T`), which includes every `.ps1`/`.bat`. The backup scheduled task runs `Invoke-WallBoardBackup.ps1` from that tree with `-RunLevel Highest` and no explicit principal (so it runs elevated as the registering admin). Any code running as the non-admin kiosk user can overwrite that script and have it executed elevated within 6 hours — arbitrary code as a high-privilege token.
**Fix:** scope the `icacls` grant to only what the updater must replace (`server\dist`, `client\dist`, `shared\dist`, `server\package*.json`); keep `scripts\` and root `.bat`/`.ps1` non-writable by the kiosk user. Alternatively run the backup task as the limited kiosk user, not `-RunLevel Highest`.

### H3 — Device-code sign-in screen self-destructs (kiosk setup is flaky/broken)
`client/src/components/AuthSetup.tsx:85-90` · `client/src/App.tsx:338` · `client/src/hooks/useAuth.ts:17-18` · `server/src/routes/auth.ts:26-28`

`AuthSetup`'s effect re-runs on `[startAuth]`; `startAuth`'s `useCallback` depends on `onAuthenticated`, which App passes as an inline arrow (`onAuthenticated={() => setIsAuthenticated(true)}`) — a new function on every App render. `useAuthStatus` returns `{...q}`, subscribing to all React-Query fields, so App re-renders on every 3-second auth poll. Each re-render → effect cleanup runs `stopPolling()` (kills the live code's countdown/poll) → `startAuth()` re-POSTs `/api/auth/start` → server returns **409 "already in progress"** → the QR/code screen is replaced by "Server error: 409", and Try Again keeps 409-ing until the 15-minute flow expires.
**Fix:** wrap `onAuthenticated` in `useCallback`; start the flow in a mount-only effect and hold `onAuthenticated` in a ref; return only needed fields from `useAuthStatus`; treat a 409 from `/api/auth/start` as "flow already pending," not an error.

### H4 — One transient unauthenticated poll bounces the kiosk into a new device-code flow
`client/src/App.tsx:147-164, 309-340`

The 4-poll debounce comment only guards the imperative `navigate('/setup')`. But line 149 sets the route's auth state to the **raw, un-debounced** poll value every tick, and the `/` route renders `<Navigate to="/setup">` off that state. A single `authenticated:false` poll (server restart, late token-store load) immediately mounts `AuthSetup`, which POSTs `/api/auth/start` and begins a real device-code flow server-side; combined with H3 the kiosk then 409-locks for up to ~15 minutes.
**Fix:** route on a debounced auth value (only flip to `false` after `unauthCountRef >= 4 || needsReauth`), and/or start the flow only on explicit user action when arriving from an authenticated session.

## MEDIUM

### Security / trust boundary

- **M1 — Graph API path injection via caller-supplied IDs.** `server/src/graph/sharepoint.ts:109,133` interpolate `siteId`/`driveId` from `req.query`/`req.params` (`routes/sharepoint.ts:63-140`) straight into `.api(\`/sites/${siteId}/drives\`)` with no validation. A crafted `siteId=x/../../me/messages` can redirect the kiosk's delegated token to other Graph GET endpoints within its scopes. **Fix:** allowlist-validate both IDs (reject `/` and `..`), prefer `encodeURIComponent`.
- **M2 — Graph-backed read routes aren't behind the admin gate.** `/api/events`, `/api/calendars`, `/api/sharepoint/*` (routes/events.ts:41, calendars.ts:8, sharepoint.ts:13) gate only on `isAuthenticated()` (a server boolean), not `requireAdminToken`. If `TRUST_LOCALHOST=false` is set for LAN exposure, board/storage correctly demand the token but these still return live M365 calendar and file listings to any LAN client. **Fix:** require the token (or a read token) on these when localhost trust is off; at minimum document the gap.
- **M3 — `xlsx` parser reachable by any local process.** `routes/board.ts:201` → `boardService.ts:385` `XLSX.read` on attacker-controlled bytes; with default `TRUST_LOCALHOST=true` the localhost bypass means *any* local process (not just the kiosk browser) can POST a crafted `.xlsm` with no token → ReDoS/proto-pollution on the single-threaded server. Bounded (LAN-only) but worse than "authenticated import parses it." **Fix:** tracked (xlsx replacement); interim — parse in a worker with a timeout.
- **M4 — Unauthenticated update/DoS trigger if `BIND_HOST` is widened.** `POST /api/update/run` is "admin-gated," but `requireAdminToken` treats any loopback `req.ip` as authenticated. Setting `BIND_HOST=0.0.0.0` (or a reverse proxy forwarding a loopback address) silently opens every admin route, including update. **Fix:** if `BIND_HOST` is non-loopback, refuse to start (or force `trustLocalhost=false`) unless `ADMIN_TOKEN` is set.
- **M5 — `multer` pinned to deprecated 1.x with unpatched DoS advisories.** `server/package.json` `^1.4.5-lts.1`. A crafted multipart upload to `/api/board/import` can crash the server → tray watchdog restart-loop. `busboy` is current, so CVE-2022-24434 isn't present. **Fix:** upgrade to `multer@^2` (the memory-storage + 20 MB usage ports directly).

### Data integrity (contract gaps vs VRSI-WALLBOARD-RULES.md §7)

- **M6 — Restore conflicts have no resolution path (§7.5 unimplemented).** `localProvider.ts:652-661` returns conflicts, `routes/storage.ts:105-114` returns 409, and there's no resolve/force endpoint. Because the conflict window compares two fixed timestamps, a flagged conflict never clears on its own — the backup is effectively unrestorable until the user edits the live job. **Fix:** add a restore option accepting per-job resolutions, audit-log each.
- **M7 — Import is three separate transactions.** `applyBoardImport` (boardService.ts:595-663) runs `writeBoardState`, then `saveJobsFile`, then prune as independent transactions. A crash mid-import leaves new statuses/ops-notes applied against the old jobs table. Self-heals on next import, but violates write atomicity. **Fix:** wrap apply+save+prune in one SQLite transaction.
- **M8 — Restore resurrects user-deleted notes and pruned state (no tombstones).** `deleteNote` hard-removes; restore's `INSERT OR IGNORE` (localProvider.ts:841-855) re-adds any backup note whose id isn't live. A note deleted on purpose reappears after an unrelated restore. **Fix:** deletion tombstones consulted by `_mergeFromBackup`.
- **M9 — Ops Schedule note never removed when the spreadsheet cell is cleared (§7.4 half-done).** `applyBoardImport` iterates only present notes (boardService.ts:625); a cleared cell leaves the old ops note displayed as current. **Fix:** strip existing `system:ops-schedule` notes for imported jobs absent from `importedNotes`.
- **M10 — Schema drift: the `deleted: true` soft-delete flag (§3) doesn't exist.** No column, no field anywhere; the code substitutes preserve-in-place. Notes are protected, but every other delete is hard. Per §18, implement the flag or get human approval to amend §3.

### Client reliability

- **M11 — "Apply all" failures on job cards are silent.** `useBoard.ts:90-160` mutations define no `onError`; `JobCard.tsx:135-162` renders none. A failed status/ship-date PATCH leaves the wall board showing the old value with no message. **Fix:** surface a card-level error line like `noteActionError`.
- **M12 — Committing a search or toggling filters discards un-applied card edits.** `key={search}` and the My/All/New/PM-MM filters unmount cards (clearing the dirty flag) without `confirmDiscardUnsaved()`. Half-edited ship date + Enter on a search → draft evaporates, no warning. **Fix:** gate those paths behind the discard guard, or lift pending edits into the store keyed by job number.
- **M13 — 3 AM watchdog can strand the kiosk.** `App.tsx:240-253` calls `window.location.reload()` with no health probe — if the server is briefly down (backup/update window) the kiosk lands on the browser error page with no retry. Compounded by `useEvents`' fetch window capturing `now` in a memo that only re-anchors on reload (event horizon freezes). **Fix:** `fetch('/health')` with retry before reloading; add a day-stamp tick to the `useEvents` memo deps.
- **M14 — Calendar/auth/SharePoint calls bypass the `{ data }/{ error }` envelope.** `calendarApi.ts`, `authApi.ts`, `sharepointApi.ts` return raw JSON and throw generic strings; server routes `events.ts:149`/`calendars.ts:28` emit bare arrays. Real server error messages never reach the UI. **Fix:** envelope those routes and route the client through `unwrap`.
- **M15 — Uninstall data-wipe ignores `DATA_DIR`/`BACKUP_DIR` overrides.** `Uninstall-WallBoard.ps1:79` hardcodes `C:\ProgramData\VRSIWallBoard`; an install that overrode `DATA_DIR` has its real DB/backups silently orphaned while an unrelated path is deleted. **Fix:** resolve the dirs via `_common.ps1` helpers.
- **M16 — Git-path updater has no rollback on failed build.** `Update-WallBoard.ps1:169-242` stops the server then pulls/rebuilds with no snapshot; a bad build restarts a half-built tree with no way back. (Dev/git installs only — release kiosks have snapshot+rollback.) **Fix:** record `git rev-parse HEAD`, `git reset --hard` + rebuild on failure.
- **M17 — Duplicate global Ctrl+S/Ctrl+F handlers.** `App.tsx:187-233` and `Dashboard.tsx:131-138` both bind document-level; Dashboard's re-opens Settings in the same keystroke App closes it, and it has no INPUT/TEXTAREA guard. **Fix:** delete Dashboard's duplicate.

## LOW (condensed)

**Server/scripts:** L1 update.ts builds a PowerShell command string (no request input today, but fragile — pass args as an array); L2 device `userCode` logged at info (exportable via logs; drop to debug); L3 stale/incorrect "no auth" comment on `board.ts:144`; L4 unquoted `powershell.exe`/`conhost` path fragments in launcher strings; L5 SHA256 verify silently skipped when a release omits the sidecar (make mandatory); L6 TOCTOU between checksum and extract; L7 encrypted token sidecar lands in the broadly-writable backups dir; L8 CI has no ESLint step despite the `any`/`as` ban and doesn't lint root `.bat`.

**Client:** L1 single-letter shortcuts fire with a `<select>` focused (guard misses `SELECT`/`contentEditable`); L2 unguarded cast of persisted role (`appStore.ts:43`) — brief "super sees everything" window on stale localStorage; two disagreeing "super user" definitions (`JobListView` vs `agendaFilter.ts`); L3 refresh-interval 0 disables polling; L4 a handful of unjustified inline styles (`SettingsPanel`/`FileBrowserPanel` z-index, `NextEventBadge` rgba, `CalendarView` height → Tailwind); L5 pragmatic `as never`/cast sites; L6 `react-big-calendar` declared with a caret (`^1.13.1`) despite TwoWeekView's "pin exact" contract — an `npm update` can break the internal `lib/Month` import; L7 fire-and-forget presence calls (unhandled rejections while server down); L8 weather fetched browser-side from open-meteo (bypasses the audit log / egress-restricted LANs); L9 recent-files errors render as "empty"; L10 `navigate('/')` without `replace` piles history; L11 ErrorBoundary can crash-loop on bad persisted state; L12 Settings Escape vs Discard leave inconsistent draft state.

**Also:** import result counter still misses the new statuses (parts_on_order/design/build show 0 — known, display-only); JSON import path drops `description` (localProvider `importJobs` is a divergent second import path, currently uncalled); rules §1 says `VERSION = 1.1.2` while all package.json are 1.1.3.

## Recommended order of work

1. **H1** — one-line guard, prevents silent data loss. Fix now.
2. **H3 + H4** — same auth subsystem; fix together to make kiosk setup reliable.
3. **H2** — scope the `icacls` grant; closes the local EoP.
4. **M4 + M2 + M1** — the "widen the bind host / disable localhost trust" cluster; do before any LAN exposure.
5. **M11–M14, M17** — client reliability/visibility quick wins for the shared touch board.
6. Data-integrity contract gaps **M6–M10** — schedule into the network-readiness project (M10 needs human sign-off to amend §3).

Nothing here is a critical remote-code vector on the LAN-only kiosk as shipped. The two data-loss items (H1, and the M-series restore gaps) and the two auth-flow highs (H3/H4) are the ones that bite in normal operation; H2 matters the moment the kiosk is multi-user or the session is otherwise reachable.
